#### 为什么购物车用Redis不用MySQL？

- 读写频繁（每次加购/改数量都写），MySQL扛不住
- 购物车数据允许丢失（不是核心数据），Redis性能高
- Hash结构天然适合：key=用户，field=SKU，value=商品信息
- 没有Redis→DB的同步负担

## 二、高并发核心技术

### 1. 分布式锁扣库存

**问题**：1000个人同时抢最后1件商品，不加锁会超卖。

**方案**：Redisson分布式锁（基于Redis的SET NX EX原子操作）

**实现位置**：[OrderServiceImpl.java:95-106](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L95-L106)

```java
// 按skuId排序 → 保证所有线程加锁顺序一致 → 防止死锁
cartItems.sort(Comparator.comparing(item -> Long.valueOf(item.get("skuId").toString())));

for (Map<String, Object> item : cartItems) {
    RLock lock = redissonClient.getLock("stock:" + skuId);
    // tryLock(等待3秒, 锁持有5秒自动释放, 时间单位秒)
    if (!lock.tryLock(3, 5, TimeUnit.SECONDS)) {
        throw new BusinessException("抢购繁忙，请重试");
    }
    acquiredLocks.add(lock);
}
// ... 扣库存 ...
// finally { releaseLocks(acquiredLocks); }
```

**答辩要点**：
- **为什么按skuId排序？** → 防死锁。线程A锁{1,2}，线程B锁{2,1}，同时加锁就死锁了。排序后大家都按{1,2}顺序加锁。
- **tryLock的3秒和5秒含义？** → 3秒是等待超时（拿不到锁等3秒），5秒是锁自动释放时间（防止线程挂了锁永远不释放）
- **MySQL层还有兜底吗？** → 有！`UPDATE t_sku SET stock=stock-? WHERE id=? AND stock>=?`，affected rows=0表示库存不足，双重保障。

### 2. 两级缓存架构

**架构图**：
```
请求 → L1 Caffeine(本地JVM,纳秒) → L2 Redis(网络IO,毫秒) → MySQL(磁盘,10ms+)

```

**实现位置**：[ProductServiceImpl.java](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java)

**防三大缓存问题**：

|------|------|-----------|----------|
| **缓存穿透** | 查不存在的数据，绕过缓存打DB | ①布隆过滤器前置拦截 ②空对象缓存3分钟 | [BloomFilterConfig.java](src/main/java/com/cloudmall/common/config/BloomFilterConfig.java) + [ProductServiceImpl.java:138-143](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java#L138-L143) |
| **缓存击穿** | 热点key过期，大量请求同时打DB | 分布式锁互斥重建，只有一个线程查DB | [ProductServiceImpl.java:118-155](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java#L118-L155) |
| **缓存雪崩** | 大量key同时过期 | Redis TTL加随机偏移（30~35分钟随机） | [ProductServiceImpl.java:80-83](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java#L80-L83) |

**答辩话术**：
> "我们用了两级缓存，L1是Caffeine本地缓存（纳秒级），L2是Redis（毫秒级）。针对三大缓存问题都有对策：穿透用布隆过滤器+空对象短缓存；击穿用Redisson分布式锁互斥重建；雪崩用TTL加随机值。"

### 3. 布隆过滤器防缓存穿透

**原理**：多个哈希函数映射到位数组。判断一个ID是否存在：
- 返回"不存在" → **100%准确**，直接返回，不打DB
- 返回"可能存在" → 有3%误判率，继续走正常流程

**实现位置**：[BloomFilterConfig.java](src/main/java/com/cloudmall/common/config/BloomFilterConfig.java)

```java
// 初始化：预计10万商品，误判率3%
bloomFilter.tryInit(100_000L, 0.03);

// 查询时先过布隆过滤器
if (!bloomFilter.mightContain(productId)) {
    return null; // 一定不存在，直接返回
}
// 可能存在 → 继续查缓存+DB
```

**答辩要点**：
> "为什么用布隆过滤器而不是直接缓存空对象？因为攻击者如果遍历100万个不存在的商品ID，缓存空对象每个也要占用Redis内存（~50字节），100万个就是50MB。布隆过滤器10万个ID只要~1MB，内存效率高出几十倍。误判率3%可以接受——即使误判穿透到DB，还有空对象缓存兜底。"

| `productQuery` | 降级 | 慢调用>30%(RT>300ms)→降级 | [L78-84](src/main/java/com/cloudmall/common/config/SentinelConfig.java#L78-L84) |

**熔断器三种状态**（答辩必问）：

```
CLOSED(关闭) ──异常率>50%──▶ OPEN(打开) ──10秒冷却──▶ HALF-OPEN(半开)
    ▲                                                    │

```

**答辩话术**：
> "限流和熔断的区别：限流是'你来得太多了，我拒绝你'，熔断是'下游已经出错了，暂时别来了'。Sentinel的断路器有三种状态——关闭→打开→半开→关闭的循环，半开状态会放一个请求试探，成功了就关闭断路器恢复正常。"

### 6. 幂等性保证

**场景**：用户手抖点了两次"提交订单"，或者网络超时重试。

**方案**：AOP + Redis SETNX

**实现位置**：
- 注解：[Idempotent.java](src/main/java/com/cloudmall/common/annotation/Idempotent.java)
- AOP：[IdempotentAspect.java](src/main/java/com/cloudmall/common/aspect/IdempotentAspect.java)
- 支付回调幂等：[OrderServiceImpl.java:210-211](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L210-L211)

**原理**：
```
首次请求: 前端传 idempotentKey=uuid-1234
         → Redis SETNX idempotent:uuid-1234 = "1" (TTL 10秒)
         → 返回true → 放行执行业务

重复请求: 前端传同样的 idempotentKey=uuid-1234
         → Redis SETNX idempotent:uuid-1234 返回false(key已存在)
         → 抛出"请勿重复提交"
```

**支付回调的幂等**：`if (order.getStatus() == 1) return;` — 已支付就直接返回，不重复处理。

## 四、高频答辩问题速查

### Q1: 下单流程中如何保证不超卖？

**答**：双层防护。
1. **Redis分布式锁** — Redisson RLock，对每个SKU加锁，同一时刻只有一个线程能扣该SKU的库存
2. **MySQL原子UPDATE兜底** — `UPDATE t_sku SET stock = stock - ? WHERE id = ? AND stock >= ?`，affected rows=0表示库存不足

代码在 [OrderServiceImpl.java:95-132](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L95-L132)

### Q2: 如何防止死锁？

**答**：所有线程按相同的顺序加锁 — 下单前把购物车商品按skuId排序（[L77-78](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L77-L78)），这样线程A和线程B不管原来顺序如何，最终都按{小→大}的顺序加锁。

### Q3: 分布式锁的可靠性如何保证？

**答**：
- **自动释放** — tryLock设置5秒leaseTime，即使线程崩溃锁也不会永久持有
- **finally释放** — 正常执行完在finally中释放
- **isHeldByCurrentThread检查** — 只释放自己持有的锁，不释放别人的

代码在 [OrderServiceImpl.java:526-534](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L526-L534)

### Q4: 缓存穿透、击穿、雪崩的区别和解决方案？

**答**：
| 问题 | 原因 | 方案 | 代码 |
|------|------|------|------|
| 穿透 | 查不存在的数据 | 布隆过滤器 + 空对象缓存 | [BloomFilterConfig.java](src/main/java/com/cloudmall/common/config/BloomFilterConfig.java) |
| 击穿 | 热点key过期 | 分布式锁互斥重建 | [ProductServiceImpl.java:118-155](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java#L118-L155) |
| 雪崩 | 大量key同时过期 | TTL加随机值 | [ProductServiceImpl.java:80-83](src/main/java/com/cloudmall/service/impl/ProductServiceImpl.java#L80-L83) |

### Q5: 秒杀为什么不在MySQL扣库存？

**答**：MySQL UPDATE操作是行锁，秒杀场景QPS上万时行锁排队成为瓶颈。Redis DECR是内存操作，单机10万QPS。我们用Redis预减库存，然后异步队列慢慢落库。代码在 [SeckillService.java:62-82](src/main/java/com/cloudmall/service/impl/SeckillService.java#L62-L82)

### Q6: 限流和熔断的区别？

**答**：
- **限流**：流量太大，超QPS阈值直接拒绝，保护系统不过载
- **熔断**：下游已经出错了（异常率高/响应慢），暂时切断请求，等下游恢复后再试探

代码在 [SentinelConfig.java](src/main/java/com/cloudmall/common/config/SentinelConfig.java)

### Q7: 如何保证幂等性？

**答**：
- **下单幂等** — 前端生成UUID作idempotentKey，后端Redis SETNX原子校验（[IdempotentAspect.java](src/main/java/com/cloudmall/common/aspect/IdempotentAspect.java)）
- **支付回调幂等** — 检查订单状态，已支付直接return（[OrderServiceImpl.java:210-211](src/main/java/com/cloudmall/service/impl/OrderServiceImpl.java#L210-L211)）

### Q8: 消息推送怎么实现的？

**答**：WebSocket + Redis Pub/Sub。
- 用户建立WebSocket连接 → JWT鉴权 → 维护userId→Session映射
- 发送消息 → Redis Pub/Sub发布到 `message:{userId}` 频道
- 订阅者收到 → WebSocket推送到浏览器
- **为什么用Redis Pub/Sub？** 多实例部署时实现跨实例消息路由

代码在 [MessageWebSocketHandler.java](src/main/java/com/cloudmall/common/handler/MessageWebSocketHandler.java)

### Q9: 购物车为什么用Redis而不是MySQL？

**答**：购物车读写极频繁（每次加购/改数量/勾选都写），MySQL扛不住这个写入量。Redis Hash结构天然适合：key=用户ID，field=SKU_ID，value=商品JSON。允许丢失（非核心数据），优先保证性能。

代码在 [CartServiceImpl.java](src/main/java/com/cloudmall/service/impl/CartServiceImpl.java)

### Q10: 内容审核怎么做的？

**答**：DFA算法（确定性有穷自动机）。
- 把敏感词库构建成树形状态机（Trie树变种）
- 时间复杂度O(n)，n=文本长度，不随词库大小增长
- 比正则匹配快几个数量级

代码在 [SensitiveWordFilter.java](src/main/java/com/cloudmall/common/utils/SensitiveWordFilter.java)

### Q11: JWT和Session的区别？为什么选JWT？

**答**：
- Session：服务端存储，有状态，无法水平扩展（需要Session共享）
- JWT：客户端存储，无状态，天生支持分布式
- 本项目是前后端分离+Vue，选JWT更合适
- 每次请求带 `Authorization: Bearer {token}`，拦截器解析出userId

代码在 [JwtInterceptor.java](src/main/java/com/cloudmall/common/config/JwtInterceptor.java) + [JwtUtil.java](src/main/java/com/cloudmall/common/utils/JwtUtil.java)

### Q12: 如果Redis挂了怎么办？

**答**：
- 购物车 → 暂时不可用（非核心）
- 分布式锁 → 扣库存降级为纯MySQL行锁
- 缓存 → 全部走DB，性能下降但功能可用
- 秒杀 → 不可用（秒杀强依赖Redis库存计数）
