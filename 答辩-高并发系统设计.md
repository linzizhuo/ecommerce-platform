# 高并发系统设计 — 答辩讲解

## 一句话概括

> 高并发的核心思路是**流量漏斗**：每一层都尽量挡住请求，能不给数据库的绝对不给数据库。

---

## 2. 限流层 — @RateLimit 注解

### 代码位置
- 注解：`common/annotation/RateLimit.java`
- 切面：`common/aspect/RateLimitAspect.java`

### 做了什么
基于 Redisson 的 **令牌桶算法**，给下单接口加了限流：

```java
@RateLimit(qps = 50, timeout = 2)  // 每秒最多50个请求
public Map<String, Object> create(...) { ... }
```

### 能讲什么
- **为什么要限流？** 防止恶意刷单、突发流量打垮数据库
- **令牌桶原理：** 每秒往桶里放50个令牌，请求来了取一个，取不到就拒绝
- **和漏桶的区别：** 漏桶匀速处理，令牌桶允许突发（令牌攒着可以一次用完）

### 老师可能会问
> "限流放在哪一层？"

答：Nginx也可以限流（`limit_req_zone`），我们是应用层限流，更灵活——不同接口可以设置不同的QPS。

## 3. 缓存层 — 两级缓存（重点讲）

### 代码位置
- 配置：`common/config/CacheConfig.java`
- 实现：`service/impl/ProductServiceImpl.java`

### 架构
```
请求 → Caffeine本地缓存(L1) → Redis缓存(L2) → MySQL数据库(L3)
          纳秒级                毫秒级          几十毫秒
          1000条                GB级           TB级
```

### 三个缓存问题 + 解决方案

| 问题 | 场景 | 我的代码怎么解决的 |
|------|------|-------------------|
| **缓存穿透** | 查不存在的id，每次都打DB | 空对象也缓存（`Product.id=-1`标记），3分钟过期 |
| **缓存击穿** | 热点key过期瞬间，大量请求打DB | Redisson分布式锁互斥重建，只有1个线程查DB |
| **缓存雪崩** | 大量key同时过期 | Redis TTL加随机偏移：`30分钟 + random(0~4)分钟` |

### 关键代码（答辩可以展示）

```java
// 缓存穿透：空对象也缓存
if (dbProduct == null) {
    Product nullMarker = new Product();
    nullMarker.setId(-1L);  // 标记为空
    redisTemplate.opsForValue().set(key, nullMarker, Duration.ofMinutes(3));
    return null;
}

// 缓存击穿：分布式锁互斥重建

if (lock.tryLock(1, 10, TimeUnit.SECONDS)) {

    // ...只有抢到锁的线程去查DB

// 缓存雪崩：随机TTL
private Duration randomTtl(long baseMinutes) {
    int offset = ThreadLocalRandom.current().nextInt(5);
    return Duration.ofMinutes(baseMinutes + offset);
}
```

### 能讲什么
- **Cache-Aside模式**：读的时候查缓存→没命中查DB→回写缓存；写的时候先更新DB再删缓存
- **为什么用两级？** 本地缓存比Redis快1000倍（无网络IO），但容量有限
- **什么数据适合本地缓存？** 商品分类这种几乎不变、访问量极大的数据
- **什么数据适合Redis？** 商品详情、库存这种需要多实例共享的

### 老师可能会问
> "缓存和数据库一致性问题怎么解决？"

答：用的是Cache-Aside模式。下单后异步删除商品缓存（`invalidateCachesAsync`），下一次查询会从数据库回源。允许短暂不一致（最终一致性），因为电商场景对实时一致性要求不高，用户看到库存差1-2个无所谓，只要不超卖就行。**分布式锁保证了不超卖，缓存只影响展示速度，不影响数据正确性。**

## 4. 分布式锁层 — 库存扣减（核心难点）

### 代码位置
- Mapper：`mapper/SkuMapper.java`（原子UPDATE）
- Service：`service/impl/OrderServiceImpl.java`

### 原始代码的问题

```java
// 问题1: 不排序 → 可能死锁
// 线程A: 先锁sku1再锁sku2
// 线程B: 先锁sku2再锁sku1
// → 互相等待对方的锁

// 问题2: 用 selectById + updateById → 竞态条件
Sku sku = skuMapper.selectById(skuId);      // 线程A读到 stock=10
sku.setStock(sku.getStock() - quantity);     // 线程B也读到 stock=10
skuMapper.updateById(sku);                   // 两个都写成9 → 超卖！
```

### 改进后的代码

```java
// 改进1: 按skuId排序，所有线程加锁顺序一致 → 防死锁
cartItems.sort(Comparator.comparing(
    item -> Long.valueOf(item.get("skuId").toString())));

// 改进2: 批量获取所有锁，任一失败释放所有 → 原子性
List<RLock> acquiredLocks = new ArrayList<>();
try {
    for (item : cartItems) {
        RLock lock = redissonClient.getLock("stock:" + skuId);
        if (!lock.tryLock(3, 5, TimeUnit.SECONDS)) {
            throw new BusinessException("抢购繁忙");
        }
        acquiredLocks.add(lock);
    }
    // ... 全部获取成功才继续
} finally {
    // 改进3: finally释放所有锁，异常也能释放 → 防死锁
    releaseLocks(acquiredLocks);
}

// 改进4: 原子UPDATE，数据库行锁兜底
@Update("UPDATE t_sku SET stock = stock - #{quantity} 
         WHERE id = #{skuId} AND stock >= #{quantity}")
int deductStock(...);
// 返回0 → 库存不足，返回1 → 扣减成功
```

### 能讲什么
- **为什么分布式锁？** 多实例部署时，Java的 `synchronized` 只能锁本进程，Redis锁是跨进程的
- **为什么还要数据库原子UPDATE？** 双重保险。Redis锁可能超时自动释放，但数据库行锁是MySQL保证的
- **为什么排序？** 死锁四个条件之一"环路等待"，排序打破了环路
- **Redisson锁原理：** Lua脚本实现，SET NX PX + Redis的pub/sub通知等待者

### 老师可能会问
> "Redisson的看门狗机制是什么？"

答：锁默认30秒过期，看门狗每10秒续期一次，防止业务没执行完锁就过期了。但我设置了`leaseTime=5秒`，所以看门狗不生效，锁5秒后自动释放——这是为了快速失败，避免长时间占锁。

## 5. 异步化 — 非关键路径不阻塞主流程

### 代码位置
- 配置：`common/config/AsyncConfig.java`
- 使用：`OrderServiceImpl` 中的 `@Async("orderExecutor")`

### 做了什么
下单成功后，两件事异步做：
1. **失效缓存**（`invalidateCachesAsync`）— 更新销量后删除商品缓存
2. **发送通知**（`sendOrderNotificationAsync`）— 短信/推送，不阻塞用户拿结果

### 能讲什么
- **同步 vs 异步：** 同步：用户等200ms拿结果。异步：用户等100ms拿结果，另外100ms后台做。
- **线程池参数：** 核心10线程常驻，最大50线程应对高峰，1000任务排队，满了让调用线程自己做（CallerRunsPolicy，不丢任务）

## 6. 连接池调优

### 代码位置
- `application.yml`

### 改了哪些
```yaml
# HikariCP — MySQL连接池
hikari:
  maximum-pool-size: 50    # 默认10→50（高并发不够用）
  minimum-idle: 10         # 预热10个等着
  connection-timeout: 3000 # 3秒拿不到就报错，别让用户傻等

# Lettuce — Redis连接池
lettuce:
  pool:
    max-active: 50
    min-idle: 5
    max-wait: 3000ms
```

### 能讲什么
- **连接池是什么？** 数据库连接用完不销毁，放回池里复用。没有池的话每次请求都要TCP三次握手+四次挥手，高并发下就是瓶颈
- **最大连接数怎么定？** 看数据库最大连接数（默认151），留一半给管理工具和其他服务

## 8. 答辩常见问题速答

**Q: 你为什么选Redis而不是直接用数据库行锁？**
A: Redis锁是**非阻塞**的（tryLock可以设超时），数据库行锁是阻塞的。高峰期锁等待会导致数据库连接池耗尽。Redis在内存中操作，比数据库锁快两个数量级。

**Q: 两级缓存的一致性怎么保证？**
A: 本地缓存靠TTL自动过期（5分钟），Redis缓存在数据变更时主动删除。允许短暂不一致——电商场景下用户看到库存差1-2个不影响体验，只要**下单时数据库行锁保证不超卖**就行。

**Q: 如果Redis挂了怎么办？**
A: 代码里有兜底——Redis挂了会自动走数据库。限流器不存在时会自动创建（trySetRate），分布式锁获取失败会抛异常回滚事务。系统**降级但不崩溃**。

**Q: 你这个系统能支撑多少并发？**
A: 限流设了50 QPS，实际Tomcat默认200线程 + 连接池50 + Redis性能远超此。瓶颈在MySQL，单表UPDATE大概1000~2000 QPS。要更高需要分库分表或引入消息队列削峰。

**Q: 秒杀场景怎么设计？**
A: 数据库里有 `t_seckill_session` 表预留了。核心思路：①Redis预减库存（大部分请求在Redis层就返回"售罄"）；②请求落到队列慢慢处理；③前端按钮置灰+验证码防脚本。这是一个渐进式的架构演进。

## 9. 一句话总结（答辩结束时说）

> 高并发不是某一个技术，是一层一层的漏斗——**前端缓存挡静态资源，限流挡恶意请求，本地缓存挡重复查询，Redis挡数据库读压力，分布式锁+原子UPDATE保证写安全，异步化让用户快点拿到结果。** 每一层各司其职，合在一起就是一个高并发系统。
