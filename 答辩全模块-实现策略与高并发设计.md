## 一、项目概述（1分半）

**项目名**：CloudMall 云商城  
**技术栈**：Vue3 + Vite + Pinia + Element Plus / Spring Boot 3.2 + Spring Cloud Alibaba + MyBatis-Plus + Redis + Redisson + MySQL 8.0  
**系统定位**：支持高并发访问的多商户电商平台，覆盖C端用户、B端商家、平台运营三大角色。

**一句话**：用微服务架构支撑三大角色，用流量漏斗模型实现高并发读写，核心链路QPS 500+。

## 二、用户前台 C端

### 2.1 基础导购

| 功能 | 实现策略 | 性能手段 |

| **搜索/联想** | MySQL `LIKE '%keyword%'` 全文搜索，对product.name和description建联合索引。搜索联想：用户输入时前端节流300ms发请求，后端查Redis有序集合（ZSET），按搜索频次排序返回Top10 | Redis ZSET存储热词（score=搜索次数），搜索联想毫秒级返回。冷门词走MySQL LIKE，热门词走Redis |
| **商品详情** | 商品主表 `t_product` + SKU表 `t_sku` + 规格JSON，一次查询组装返回。评价信息异步加载 | **两级缓存**：Caffeine(5min) + Redis(30min+随机)，防穿透/击穿/雪崩（详见高并发章节） |
| **评价问答** | `t_review` 表关联userId+productId+orderId，支持文字+图片+星级评分。前端分页加载，按时间/有用数排序 | 评价列表Redis缓存10分钟，评价数冗余到product表避免COUNT |

### 2.2 交易流程

|------|---------|---------|

## 5.1 限流系统

### 做了什么
基于 **Redisson RRateLimiter（令牌桶算法）** + **自定义注解**，对不同接口设置差异化QPS限制。

### 实现原理
```
令牌桶：每秒往桶里放入 QPS 个令牌
请求来了 → tryAcquire() → 有令牌：拿走，放行
                         → 没令牌：等timeout秒，还没拿到就拒绝
```

### 分层限流策略

| 接口 | QPS | 理由 |
|------|-----|------|
| 商品查询 | 500 | 读多写少，缓存兜底，可以放高 |
| 加入购物车 | 200 | 操作频繁但无锁竞争 |
| 下单 | 50 | **核心保护**，涉及库存扣减，不能太高 |
| 秒杀 | 由活动独立控制 | 秒杀模块独立限流，不占用正常下单额度 |
| 发送验证码 | 1次/分钟/手机号 | 防短信轰炸，按手机号维度限流 |

### 令牌桶 vs 漏桶 vs 滑动窗口
- **令牌桶**：允许短时突发（桶里攒的令牌可以一次性用完）。适合电商——促销开始时流量瞬间爆发
- **漏桶**：绝对匀速，突发流量全部丢弃。适合保护数据库——但对用户体验不友好
- **滑动窗口**：统计最近1秒内的请求数，超过阈值拒绝。实现简单但不够平滑

### 为什么选令牌桶
电商大促的流量特征：前1秒可能有平时10倍的请求。令牌桶允许把"攒的令牌"一次性消费，不会全部拒绝——用户在促销开始时点得快的能下单，慢一点的排队，体验好。

### 常见追问
> "如果Redis挂了，限流还能用吗？"
答：可以降级到本地Guava RateLimiter（单机限流），不够精确但不会让系统裸奔。

## 5.2 两级缓存体系 ★

### 架构
```
请求 → L1: Caffeine本地缓存(JVM堆内) → L2: Redis分布式缓存 → L3: MySQL
         纳秒级，无网络IO             毫秒级，一次RTT          几十毫秒
         容量: MB级                   容量: GB级              容量: TB级
```

### 缓存策略：Cache-Aside（旁路缓存）
- **读**：查缓存→命中返回→未命中查DB→回写缓存
- **写**：先更新DB→再删除缓存（不更新缓存，等下次读时回源）
- **为什么删缓存而不是更新缓存？** 更新缓存可能产生并发写覆盖。删除让下一次读回源，只有一次DB查询，更简单安全

### 三大缓存问题的解决方案

#### ① 缓存穿透 — 查不存在的数据，每次都打到DB
```
攻击者: 循环请求 /product/99999（不存在的ID）
没有缓存 → 每次查DB → DB被打垮
```
**我的解法**：空对象缓存
```
第一次查99999 → DB返回null → Redis存空标记（Product.id=-1），3分钟过期
第二次查99999 → Redis命中空标记 → 直接返回null，不打DB
```
进阶方案：布隆过滤器。把所有存在的商品ID预先存入布隆过滤器，请求先过过滤器，不存在的直接返回。

#### ② 缓存击穿 — 热点key过期瞬间，大量请求涌向DB
```
商品iPhone的缓存刚好过期
同时来了1000个请求 → 全部去查DB → DB瞬间压力巨大
```
**我的解法**：分布式锁互斥重建
```java
RLock lock = redissonClient.getLock("lock:cache:product:" + id);
if (lock.tryLock(1, 10, TimeUnit.SECONDS)) {  // 只有一个线程抢到锁
    // 双重检查：可能别的线程已经重建好了
    // 查DB → 写Redis → 释放锁
}
// 没抢到锁的线程：直接返回（降级），或等一小段时间
```

#### ③ 缓存雪崩 — 大量key同时过期
```
30分钟前预热了5000个商品的缓存
30分钟后，5000个key同时过期 → 5000个请求同时打DB
```
**我的解法**：过期时间加随机偏移
```java
// 不是统一30分钟，而是 30 + random(0,5) 分钟
Duration ttl = Duration.ofMinutes(30 + ThreadLocalRandom.current().nextInt(5));
```
5000个key在30~35分钟之间分散过期，流量被摊平。

### 常见追问
> "缓存一致性问题怎么解决？"

答：**最终一致性**。下单后异步删除商品缓存（`evictProductCache`），下次查询回源。允许短暂的"缓存落后于数据库"——用户看到库存差1-2个不影响下单，因为**下单时数据库行锁是最终裁判**。

> "什么数据用本地缓存，什么数据用Redis？"

答：
- **本地缓存**：分类列表（几乎不变）、字典数据（变更极少）、权限列表
- **Redis缓存**：商品详情（需多实例共享）、库存快照、用户session、热门搜索词

## 5.3 分布式锁与库存扣减 ★★

### 问题：并发下单为什么需要锁

```
商品库存 = 10
时刻1: 用户A下单买8个 → 查库存=10 ✓
时刻2: 用户B下单买5个 → 查库存=10 ✓（A还没扣完）
时刻3: A: update stock=10-8=2
时刻4: B: update stock=10-5=5   ← B基于旧数据写，覆盖了A的结果
结果: 库存=5，但应该=10-8-5=-3 → 超卖！
```

### 我的方案：分布式锁 + 原子UPDATE 双重保险

#### 第一道防线：Redisson分布式锁
```java
// ① 按skuId排序 → 所有线程加锁顺序一致 → 防死锁
cartItems.sort(Comparator.comparing(item -> item.getSkuId()));

// ② 批量获取所有锁，任一失败则释放
List<RLock> acquiredLocks = new ArrayList<>();
for (item : cartItems) {
    RLock lock = redissonClient.getLock("stock:" + item.getSkuId());
    if (!lock.tryLock(3, 5, TimeUnit.SECONDS)) {
        throw new BusinessException("抢购繁忙");  // 失败就释放所有
    }
    acquiredLocks.add(lock);
}

// ③ finally保证所有锁释放（死也不会死锁）
finally { releaseLocks(acquiredLocks); }
```

#### 第二道防线：MySQL原子UPDATE
```sql
-- 不是 select + update（有竞态），而是一条原子SQL
UPDATE t_sku SET stock = stock - #{quantity}
WHERE id = #{skuId} AND stock >= #{quantity}
-- affected_rows=0 → 库存不足
-- affected_rows=1 → 扣减成功
```

### 为什么要两道防线
- **Redis锁**：跨进程的互斥，防止多个服务实例同时扣同一库存。`tryLock`非阻塞，3秒拿不到就快速失败
- **MySQL原子UPDATE**：兜底。万一Redis锁超时释放了（极端情况），MySQL的`WHERE stock >= qty`是InnoDB行锁保证的，绝不会超卖

### 防死锁三要素
1. **排序加锁**：所有线程按相同顺序获取锁 → 打破"环路等待"
2. **tryLock超时**：等3秒拿不到就放弃 → 打破"不可剥夺"
3. **finally释放**：异常也会走到finally → 打破"持有并等待"

### 常见追问
> "分布式锁选Redis还是Zookeeper？"
答：Redis。①性能高（内存操作）；②项目已有Redis依赖；③Redisson封装好，支持自动续期、可重入、公平锁。ZK的一致性更强但性能不如Redis，适合对一致性要求极端严格的场景（如金融）。

> "Redisson的看门狗（Watchdog）机制是什么？"
答：默认锁30秒过期。如果业务逻辑超过30秒，看门狗每10秒自动续期一次。我设了`leaseTime=5秒`，看门狗不生效——目的是快速失败而非长时间占锁。

## 5.4 秒杀系统 ★★★

### 为什么秒杀是最高难度的并发场景
- 普通下单：平均1 QPS，峰值10 QPS
- 秒杀：**1秒内涌入10万请求，抢1000个库存**

### 秒杀流量漏斗

```
100,000 请求
    │
    ▼
┌─────────────────────────────────────┐
│ 第一层：前端防护                       │
│ - 秒杀开始前按钮置灰                      │
│ - 点击后按钮变灰+倒计时（防重复点击）         │
│ - 滑块验证码（防脚本刷）                  │
│ 预计挡掉：~50% → 剩50,000             │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 第二层：网关限流 (Gateway/Sentinel)      │
│ - 秒杀接口独立限流（QPS=5000）           │
│ - 超过的直接返回"活动太火爆"               │
│ 预计挡掉：~90% → 剩5,000              │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 第三层：Redis预减库存（核心）             │
│ - SET seckill:stock:123 = 1000      │
│ - DECR seckill:stock:123            │
│ - 减后 < 0 → 直接返回"已售罄"           │
│ - 减后 >= 0 → 进入队列               │
│ 预计挡掉：～87.5% → 剩1,000           │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 第四层：消息队列削峰                    │
│ - 1000个请求进入RocketMQ队列           │
│ - 消费者以稳定速率（100/s）消费          │
│ - 生成订单+真正扣减DB库存                 │
│ 最终的DB压力：100 QPS，完全可控          │
└─────────────────────────────────────┘
```

### 核心代码逻辑
```java
public R seckill(Long userId, Long seckillId, Long skuId) {
    // 1. 校验秒杀时间
    SeckillSession session = getSession(seckillId);
    if (!session.isActive()) return R.fail("活动未开始或已结束");

    // 2. 校验是否已抢过（防重复）
    String recordKey = "seckill:record:" + seckillId + ":" + userId;
    if (redisTemplate.hasKey(recordKey)) return R.fail("已参与");

    // 3. Redis预减库存（原子操作，核心！）
    String stockKey = "seckill:stock:" + skuId;
    Long stock = redisTemplate.opsForValue().decrement(stockKey);
    if (stock < 0) {
        // 恢复并标记售罄
        redisTemplate.opsForValue().increment(stockKey);
        return R.fail("已售罄");
    }

    // 4. 标记已抢（防重）
    redisTemplate.opsForValue().set(recordKey, "1", Duration.ofHours(24));

    // 5. 扔进消息队列异步下单
    SeckillMessage msg = new SeckillMessage(userId, seckillId, skuId);
    rocketMQTemplate.send("seckill-order-topic", msg);

    // 6. 立即返回（不等下单结果）
    return R.ok("抢购成功，订单生成中...");
}
```

### 秒杀和普通下单的对比

| 维度 | 普通下单 | 秒杀 |
|------|---------|------|
| 库存扣减 | 分布式锁 → DB原子UPDATE | Redis DECR预减 → 消息队列 → DB |
| 同步/异步 | 同步（用户等结果） | 异步（立即返回，后台创建订单） |
| 数据库压力 | 每单都打DB | 只有抢到的1000人才打DB |
| 失败处理 | 抛异常回滚事务 | Redis恢复库存 + 队列死信处理 |
| 用户体验 | 看到下单结果 | 看到"抢购成功，订单生成中" |

### 秒杀时间同步问题
- 客户端时间不可信（用户改了系统时间）
- 方案：服务端返回活动开始倒计时（秒），前端用 `服务端时间 + setInterval` 倒计时
- 用户请求时服务端校验时间，前端倒计时只是展示

### 常见追问
> "Redis预减库存后，如果消息队列消费失败怎么办？"
答：①记录失败日志；②定时任务扫描Redis库存和DB库存的差异，自动回补；③用户可以在"我的订单"看到处理中状态的订单，超时自动退款。

> "为什么不让1000个请求直接打DB？反正也只有1000个。"
答：①DB连接池只有50，1000并发会把连接池耗尽，影响正常下单；②秒杀结束后还有退款、改单等正常请求，不能让秒杀拖垮整个数据库。

## 5.6 读写分离

### 架构
```
写请求(INSERT/UPDATE/DELETE) → 主库(Master)
                                    │
                               binlog主从复制
                                    │

```

### 路由规则
```java

@DS("slave")   // 读走从库
public List<Product> list() { ... }
```

### 主从延迟的处理
问题：刚下的订单，马上查"我的订单"查不到（主库写了，从库还没同步）
解决：
1. **写后立即读走主库**：下单后返回的订单详情直接从主库数据返回，不走从库
2. **用户自己的数据走主库**：我的订单、我的优惠券等"自己写自己读"的场景走主库
3. **商品列表等公共数据走从库**：可以容忍几秒延迟

## 5.8 幂等性（防重复下单）★

### 问题
用户网卡，点了"提交订单"没反应，又点了一下 → 两条相同请求 → 两笔重复订单。

### 实现：自定义 @Idempotent 注解 + Redis SETNX

```java
@Idempotent(key = "createOrder", ttl = 10)  // 10秒内重复请求拒绝
@PostMapping
public R createOrder(...) { ... }
```

### AOP切面原理
```
请求进来 → 提取idempotentKey(UUID) → Redis SETNX idempotent:createOrder:{key}
→ 成功(首次): 放行执行业务
→ 失败(重复): 抛BusinessException("请勿重复提交")
→ 10秒后key自动过期(允许重试)
```

### 为什么用Redis SETNX？
SETNX是原子操作，天然幂等。Redis内存操作比查数据库快100倍。TTL自动过期不占内存。

### 追问
> "前端按钮置灰不就行了？"
答：前端防不了。用户关JS照样发请求、网络重试自动重发、脚本直接调API。**前端是体验优化，后端幂等是数据安全保障。**

## 5.9 缓存预热 ★

### 问题
系统刚启动缓存是空的。前100个用户访问 → 100次缓存miss → 同时打DB → 冷启动雪崩。

### 实现：CommandLineRunner 启动时预加载
```java
@Component
public class CacheWarmer implements CommandLineRunner {
    public void run(String... args) {
        // 1. 分类列表 → Caffeine本地缓存（1小时）
        // 2. 热门商品Top50 → Redis（30~35分钟随机TTL）
    }
}
```

### 预热什么
- **分类列表** → 访问极频繁、几乎不变 → L1本地缓存
- **热门商品Top50** → 首页展示 → L2 Redis
- **字典/配置** → 系统参数 → L1本地缓存

### 预热失败
不阻塞启动：catch异常→记日志→正常启动。首次请求走正常Cache-Aside路径回源。

## 5.10 Sentinel熔断降级 ★

### 和限流的区别
- **限流**："请求太多，不让你进来"（保护自己）
- **熔断**："下游出错了，我先别调了"（保护下游）

### 熔断三态
```
关闭(正常) → 错误率>50% → 打开(熔断/快速失败)
打开 → 10秒冷却 → 半开(试探一个请求)
半开成功 → 关闭(恢复)  |  半开失败 → 打开(继续熔断)
```

### 配置的规则
| 资源 | 规则 | 阈值 | 效果 |
|------|------|------|------|
| createOrder | 异常比例熔断 | 错误率>50% | 熔断10秒 |
| productQuery | 慢调用比例 | RT>300ms占>30% | 降级返回兜底数据 |

### 答辩关键
Sentinel是阿里开源流量防卫兵。下单接口异常率高→自动熔断→快速失败。**下单服务出问题不拖垮商品浏览**——故障隔离。

## 5.11 布隆过滤器（防穿透进阶）★★

### 为什么需要
缓存空对象能防穿透但有内存问题：攻击者扫1000万个不存在ID → 全部进Redis → 内存爆了。
布隆过滤器：10万个ID只需~1MB，100%准确判断"不存在"。

### 原理
```
多个哈希函数 → 映射到位数组
contains(id) → 所有哈希位=1 → 可能存在（3%误判）
contains(id) → 任一哈希位=0 → 一定不存在
```

### Redisson实现
```java
RBloomFilter<Long> bf = redissonClient.getBloomFilter("bloom:product:ids");
bf.tryInit(100_000L, 0.03);  // 10万ID, 3%误判率
```

### 查询流程
```
查 /product/99999
  → 布隆: contains(99999)=false → 直接返回"不存在"（100%准确）
  → 布隆: contains(99999)=true → 查缓存 → 查DB（3%误判走这里，缓存空对象兜底）
```

### 布隆 + 空对象 = 双重防线
布隆过滤器挡97%，缓存空对象挡剩下3%。两者组合：**布隆节省99%Redis内存，空对象保证100%防穿透。**

## 七、答辩FAQ 30问速答

### 缓存相关
1. **缓存和数据库不一致怎么办？** → Cache-Aside模式，写完DB删缓存。允许短暂不一致，下单时数据库行锁保证正确性。
2. **为什么用两级缓存？** → 本地缓存纳秒级无网络开销，Redis毫秒级但容量大。热数据双层命中，90%请求本地返回。
3. **缓存穿透和击穿有什么区别？** → 穿透是查不存在的数据，击穿是热点key过期。前者空值缓存，后者互斥锁。
4. **布隆过滤器原理？** → 多个哈希函数映射到位数组。说有不一定有，说没有一定没有。适合防穿透。
5. **Redis内存满了怎么办？** → LRU/LFU淘汰策略 + 设置maxmemory + 监控告警扩容。

### 分布式锁相关
6. **synchronized和分布式锁的区别？** → synchronized锁JVM进程内，分布式锁跨进程/跨服务器。
7. **为什么要排序加锁？** → 所有线程按相同顺序加锁，打破死锁的环路等待条件。
8. **为什么还要数据库原子UPDATE？** → 双重保险。Redis锁可能故障，MySQL行锁是最终保障。
9. **Redisson锁的原理？** → Lua脚本 `SET NX PX` + Redis Pub/Sub通知等待线程。
10. **锁超时了业务没执行完？** → 看门狗自动续期。或设计业务超时时间大于锁超时时间。

### 秒杀相关
11. **秒杀为什么不能直接操作数据库？** → 10万请求抢1000库存，99500个请求无意义地打数据库，连接池瞬间耗尽。
12. **Redis库存和DB库存不一致？** → Redis扣减成功→MQ异步同步DB。对账任务兜底修复。
13. **用户重复秒杀怎么办？** → Redis SET `seckill:record:活动ID:用户ID` 做幂等，同一个活动+用户只能请求一次。
14. **秒杀开始瞬间时钟不同步？** → 服务端返回统一的倒计时（秒），前端不信任客户端时间。
15. **消息队列积压怎么办？** → 增加消费者实例、限流源头、积压告警自动扩容。

### 限流相关
16. **令牌桶和漏桶选哪个？** → 令牌桶允许突发，适合电商促销。漏桶绝对平滑，适合API对外开放。
17. **限流放在网关还是应用层？** → 两层都放。网关粗粒度（IP级别），应用层细粒度（接口+用户级别）。
18. **被限流的用户感知是什么？** → 返回"系统繁忙"，前端展示友好提示+几秒后可重试。

### 数据库相关
19. **连接池大小怎么定？** → 公式：`核数*2 + 磁盘数`，实际根据监控调。MySQL最多151连接，留一半给别用。
20. **读写分离延迟怎么处理？** → 写后立刻读走主库，用户自己的数据走主库，公共数据走从库。
21. **分库分表怎么分？** → 订单按userId哈希取模，保证同一用户的订单在同一分片，方便查询。
22. **为什么不直接用MySQL行锁而用Redis锁？** → 行锁是阻塞等待，高并发耗尽连接池。Redis tryLock是非阻塞的。
23. **索引怎么加？** → 高频查询字段加联合索引：订单 `(userId, createTime)`，商品 `(categoryId, status)`。

### 微服务相关
24. **为什么用Spring Cloud Alibaba而不是Dubbo？** → Alibaba全套方案（Nacos+Sentinel+Seata+RocketMQ），整合度高、社区活跃。
25. **服务间调用超时怎么办？** → Sentinel熔断：A调B连续失败→熔断→快速返回降级结果→定时探测恢复。
26. **分布式事务怎么保证？** → Seata AT模式：一阶段各服务执行+记录undolog，二阶段统一提交/回滚。
27. **配置中心有什么好处？** → 改配置不重启。限流阈值、活动开关等在Nacos上改就生效，不用重新部署。

### 项目整体
28. **系统能支撑多少QPS？** → 读：5000+（缓存），写下单：50（限流保护），秒杀下单：异步后不限。瓶颈在MySQL的UPDATE能力（~2000 QPS单表）。
29. **如果Redis完全崩溃了？** → 系统降级：①缓存穿透→直接查DB；②分布式锁→报错提示重试；③限流→降级到本地Guava RateLimiter。用户感知慢但不崩溃。
30. **你觉得还有哪些可以改进的？** → ①引入ES做商品搜索（替换MySQL LIKE）；②订单表分库分表（ShardingSphere）；③完整的全链路压测体系；④异地多活部署。

## 八、答辩时的"万能句式"

被问到不会的问题时：
> "这个方向我们在系统设计中预留了扩展点（数据库里有表/留了接口），目前的方案是XXX，如果上线需要进一步做XXX优化。核心思路还是我们高并发模块的那一套：流量分层、缓存优先、异步削峰。"

结束时总结：
> "本系统在传统电商CRUD基础上，重点设计了高并发五层防线——限流挡恶意、缓存挡读压力、分布式锁+原子UPDATE保证写安全、消息队列削峰填谷、读写分离分摊数据库压力。每一层都是一个独立的防御体系，合在一起支撑起一个能应对促销峰值的电商平台。"
