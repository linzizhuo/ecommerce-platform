# CloudMall 电商平台 — 答辩准备指南

## 一、项目概述

**CloudMall** 是一个**高并发电商平台**，采用**前后端分离架构**：
- 前端：Vue 3 + TypeScript + Element Plus + ECharts + Vite
- 后端：Spring Boot 3.2 + MyBatis-Plus + Redis + MySQL
- 三端分离：**C端（用户）**、**商家端**、**平台管理端**

> 亮点：系统针对秒杀、拼团、预售等高并发场景做了专门的架构设计。

## 四、核心业务流程

### 4.1 秒杀流程（最高并发场景）

```
用户点击"立即秒杀"
     │
     ▼
┌─────────────────┐
│ 1. Redis读场次    │  ← 互斥锁防缓存击穿，miss时自动回种
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 校验时间窗口    │  ← now必须在 [startTime, endTime] 之间
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 快速库存拦截    │  ← GET stock key，售罄直接拒绝（省Lua往返）
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. Lua原子操作    │  ← 一个Redis往返完成：
│  ① 检查是否已参与  │     检查参与 + 初始化库存 + 扣库存 + 标记参与
│  ② 扣减库存       │     返回值：-1=已参与 -2=售罄 ≥0=成功
│  ③ 标记已抢       │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. 消息入队       │  ← 秒杀成功 → 消息写入Redis队列，立即返回
│  返回"订单生成中"  │      前端每1.5秒轮询一次结果（最多20次）
└────────┬────────┘
         ▼
┌─────────────────┐
│ 6. 异步消费队列   │  ← @Scheduled每500ms批量消费（最多50条/批）
│  批量创建订单     │      先搬到备份队列 → 批量写DB → 删除备份
└─────────────────┘
    消息不丢机制：如果处理中崩溃 → 下次启动从备份队列恢复
```

### 4.2 用户下单流程

```
商品详情 → 选SKU → 加购物车 → 购物车结算

```

```
    → 其他人参团 → 人数达标 → 拼团成功
```

```
    → 用户支付尾款 → 发货
```

## 五、技术亮点（答辩重点）

### 5.1 秒杀高并发方案

| 技术 | 作用 | 实现 |
|------|------|------|
| **Redis缓存** | 秒杀场次数据缓存，减轻DB压力 | `seckill:session:{id}` TTL=活动结束+10min |
| **Lua原子脚本** | 一个Redis往返完成"检查参与+扣库存+标记"，无竞态 | `seckillScript` @PostConstruct加载 |
| **互斥锁防击穿** | 缓存miss时只允许一个线程查DB回种 | `SETNX` 3秒锁 + 随机退避重试 |
| **异步下单** | 秒杀返回后异步创建订单，削峰填谷 | Redis队列 + @Scheduled消费 |
| **消息不丢** | 处理中崩溃消息可恢复 | `RPOPLPUSH` 原子搬迁到备份队列 |
| **Redisson令牌桶限流** | 接口级QPS控制 | @RateLimit注解 + AOP切面 |
| **实时库存Pipeline** | 一次Redis往返拿所有场次库存 | Redis Pipeline合并查询 |

### 5.2 限流机制 (RateLimitAspect)

```java
@RateLimit(qps = 200, timeout = 2)  // 秒杀接口：200QPS，超时2秒拒绝
@PostMapping("/{sessionId}")
public R seckill(...) { ... }
```

- 基于 **Redisson RRateLimiter**，在Redis中维护令牌桶
- AOP切面自动拦截 `@RateLimit` 注解的方法
- 超过QPS直接返回"请求过于频繁"

### 5.3 JWT认证 + 拦截器

```
请求 → JwtInterceptor → 从Header取 "Authorization: Bearer <token>"
     → 验证token有效性 → 解析出userId存入request attribute
     → Controller通过 request.getAttribute("userId") 获取
```

- 无状态认证，不依赖Session
- BCrypt加密存储密码
- 公开接口（商品浏览、搜索等）不需要登录

### 5.4 缓存策略

| 场景 | 策略 |
|------|------|
| 秒杀场次列表 | Redis缓存，10秒TTL，实时库存覆盖 |
| 单场秒杀场次 | Redis缓存，mutex防击穿，TTL=活动结束+缓冲 |
| 商品详情 | Redis缓存 |
| 类目树 | Redis缓存 |
| 搜索热词 | Redis缓存 |
| 布隆过滤器 | 防缓存穿透（用户ID、商品ID等） |

### 5.5 其他技术点

- **RBAC权限模型**: User → Role → Permission 三层，管理员/商家/用户三套菜单
- **WebSocket消息推送**: 站内消息实时通知
- **数据库连接池调优**: HikariCP max 50连接
- **Tomcat线程池**: max 200线程 + accept-count 100队列

## 七、答辩可能问到的问题

### Q1: 为什么选择Redis做秒杀？
A: Redis单线程模型，Lua脚本能保证原子性。一个Lua脚本完成"检查+扣库存+标记参与"三步操作，避免了数据库行锁的竞争。而且Redis在内存操作，单机QPS可达10万+，远高于MySQL。

### Q2: 如何防止缓存击穿？
A: 用"互斥锁"模式。当缓存miss时，多个并发请求中只有一个能拿到`SETNX`锁去查DB回种，其他线程自旋等待缓存或随机退避重试。

### Q3: 秒杀消息丢了怎么办？
A: 使用`RPOPLPUSH`（原子操作）把消息从主队列搬到备份队列。处理成功后删除备份；如果处理中途崩溃，下次启动时`recoverOrphanMessages()`把备份队列的消息搬回主队列。

### Q4: 限流是怎么实现的？
A: 通过`@RateLimit`自定义注解 + AOP切面，底层用Redisson的令牌桶算法。在Redis中维护每个接口的令牌桶，超过QPS就拒绝请求。

### Q5: 前后端分离怎么处理登录？
A: 用户登录后后端返回JWT Token，前端存localStorage，后续请求在Header带`Authorization: Bearer <token>`。后端`JwtInterceptor`拦截所有`/api/**`请求，校验token并解析出userId。

### Q6: 系统并发能力如何？
A: Tomcat 200线程 + HikariCP 50连接 + Redis连接池50 + Redisson令牌桶限流。秒杀接口限制200QPS，单机理论可支撑数百QPS。

### Q7: 为什么用MyBatis-Plus而不是JPA？
A: MyBatis-Plus对复杂SQL查询更友好，手写SQL灵活度高。秒杀等高性能场景需要精确控制SQL，ORM的自动生成反而不可控。
